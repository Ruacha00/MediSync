# MediSync 系统架构与商业模式报告

**项目名称：** MediSync — 行为自适应用药提醒系统

---

## 一、项目概述

### 1.1 问题背景

慢性病患者（高血压、糖尿病、哮喘、抗凝治疗等）普遍面临用药依从性不足的问题。现有解决方案存在明显缺陷：

- **固定时间提醒 APP（如 Medisafe、MyTherapy）：** 不适配用户真实生活节奏（通勤、会议、不规律作息、出差），导致通知疲劳，用户最终忽略提醒。
- **智能药瓶/药盒硬件（如 AdhereTech）：** 只能被动监测用药行为，无法预测和主动干预，且设备成本高、推广门槛大。

两类方案的共同问题：**缺乏对个人行为模式的理解和适应能力。**

### 1.2 解决方案

MediSync 是一款以手机为核心的 AI 驱动用药提醒应用。核心差异化在于：不使用固定时间闹钟，而是通过 AI 学习用户的日常行为模式，在用户最可能服药的时刻发送个性化提醒。

**核心功能：**

- AI 自适应提醒时间选择
- 漏服后智能重调度（基于药物安全窗口）
- 基于日历的出门提醒（提示带药/提前服药）
- 护理管理者轻量仪表盘
- EHR 双向数据集成（FHIR API）

## 二、系统架构

### 2.1 系统整体结构

MediSync 系统分为三个层级：**患者端 APP**、**MediSync 后台平台**、**外部系统对接**。

**患者端 APP** 负责用户交互、数据采集和提醒推送。**后台平台** 运行 AI 模型、管理数据、提供护理仪表盘。**外部系统** 包括 EHR（通过 FHIR API 对接）和保险公司（仅接收聚合报告）。

### 2.2 数据采集方案

MediSync 采用**两类数据源**，平衡智能化与隐私保护：

**主动反馈数据（核心数据源）：** 用户每次收到提醒后点击"已服药"、"稍后提醒"或"跳过"，系统记录响应时间和行为，AI 据此学习用户的用药习惯。该数据隐私风险低，无需额外权限。

**日历数据（辅助数据源）：** 读取用户手机日历中的日程安排（如会议、出行），用于避开忙碌时段发送提醒，以及在出门前触发带药提醒。需用户授权日历读取权限。

**未采用的数据源：** GPS 定位和屏幕解锁状态。虽然这些数据能进一步提升提醒精度，但隐私敏感度高，在答辩中难以论证合理性，且日历数据已足够支撑"出门提醒"功能。

### 2.3 运行流程

#### 阶段一：用户注册与设置

1. 用户下载 APP 并注册账号。
2. 选择药物设置方式：
   - **手动输入：** 用户自行添加药物名称、剂量、频率。
   - **EHR 自动导入：** 通过 FHIR API 连接医疗机构 EHR 系统，自动拉取当前处方和用药清单。
3. 授权日历读取权限，用于 AI 分析日程。

#### 阶段二：日常使用 — AI 智能提醒

1. **AI 选择提醒时间：** 根据用户历史服药反馈（什么时间点击"已服药"的比例最高）和日历空闲时段，选择当天最优提醒时刻，而非固定闹钟。
2. **推送提醒通知：** 在选定时间发送推送通知。
3. **用户响应分三条路径：**
   - **已服药 →** 记录成功，AI 学习数据更新，强化该时段的权重。
   - **稍后提醒 →** AI 在药物安全窗口内重新选择下一个最佳时机再次提醒。
   - **未响应/跳过 →** 进入漏服处理流程。

#### 阶段三：漏服智能重调度

漏服处理是 MediSync 的关键差异化功能，分四层逻辑：

**第一层 — 药物安全窗口（临床规则）：** 每种药物在系统中预设一个"可补服时间窗"。例如降压药漏服3小时内可补，抗凝药窗口更短。这些规则基于临床指南预设，**AI 不做临床决策**，只在安全范围内优化提醒时间。

**第二层 — AI 选择最佳补服时机：** 若仍在安全窗口内，AI 结合当前日历空闲和历史响应数据，选择用户最可能响应的时间再次提醒。

**第三层 — 用户操作：** 用户可选择"现在补服"、"稍后提醒"或"跳过本次"。

**第四层 — 超出窗口自动跳过：** 超出安全时间窗后，系统不再提醒补服，自动标记为漏服，数据记入依从性报告。

#### 阶段四：基于日历的出门提醒

系统检测到日历中的外出事件（如早上9点的会议），推断用户大约8:30需要出门，提前发送提醒："出门前记得服药/带药"。此功能无需 GPS，完全基于日历推断。

#### 阶段五：数据同步与分发

依从性数据从 MediSync 平台分三路流出：

- **护理管理者仪表盘：** 轻量级后台界面，展示患者列表和依从性概览。高风险患者（如连续漏服3次以上）标红预警，方便护士/慢病管理师重点关注。
- **EHR 回传：** 通过 FHIR API 将患者依从性数据写回 EHR，医生在现有系统中即可查看。
- **保险公司报告：** 仅提供去标识化的聚合数据（如群体依从率、住院率变化），**绝不包含任何个人信息**。

## 三、商业模式

### 3.1 收入模式：三通道结构

MediSync 采用**双通道获客 + 第三方付费**的混合商业模式，参考美国处方数字疗法（Prescription Digital Therapeutics, PDT）行业实践：

**通道一：患者自费订阅（B2C）**

- 基础提醒功能免费，AI 智能调度、日历集成等高级功能按月订阅（如 $5.99/月）。
- 目的：保证用户基数，降低使用门槛。
- 用户在 App Store 自行下载注册。

**通道二：医疗机构处方采购（B2B）**

- 医生在 EHR 中像开药一样"开具" MediSync 给患者，患者收到激活链接后免费使用全部功能。
- 费用由医疗机构按患者人数支付（SaaS 模式）。
- 此通道天然要求 EHR 集成，处方通道本身就需要与 EHR 对接。
- 行业先例：Welldoc（BlueStar）糖尿病管理平台即采用类似模式。

**通道三：保险公司批量采购（B2B）**

- 保险公司为其会员购买 MediSync 使用权，会员免费使用。
- 保险公司获得的价值：会员依从性提升 → 并发症减少 → 住院/急诊降低 → 赔付成本下降。
- MediSync 定期向保险公司提供去标识化的群体效果报告（如"使用 MediSync 的会员群体住院率下降 X%"），作为续约和 ROI 评估依据。
- 同时可作为保险公司的会员福利卖点，在竞争性市场中吸引雇主和个人选择其计划。

### 3.2 商业逻辑核心

患者依从性提高 → 病情控制改善 → 住院和急诊减少 → 医疗系统和保险公司节省大量成本。因此医院和保险方有强动力为 MediSync 付费。MediSync 的价值定位是**卖"结果"，而非"数据"**。

### 3.3 长期发展路径

创业初期可先以患者自费 + 医院直接采购（不走保险报销）启动。长期来看，如需通过保险报销通道，可能需要申请 FDA 认证或获取 CPT 报销代码。

## 四、伦理与数据隐私

### 4.1 数据隐私架构

MediSync 在架构层面将**个人数据**与**聚合报告**严格分离：

**个人层数据（严格保护）：** 包括患者的服药记录、漏服次数、行为模式、日历数据。仅患者本人、授权的护理管理者和 EHR 系统可访问。

**聚合层数据（可对外共享）：** 去标识化后的群体统计数据，如整体依从率、按病种分类的趋势、活跃用户数等。保险公司仅可访问此层级数据。

### 4.2 保险公司数据防火墙

**保险公司不得获取任何可识别个人身份的健康数据（PHI）。** 具体措施：

- 合作协议中写入数据防火墙条款，明确约定：MediSync 不得向保险公司披露任何个人级依从性数据；保险公司不得将合作中获得的任何信息用于核保、理赔决策或费率调整。
- 保险公司获得的仅为群体级效果报告，用于评估采购 MediSync 的 ROI，不涉及任何个人。
- 违反防火墙条款即构成违约，可追究法律责任。

### 4.3 HIPAA 合规

MediSync 处理患者用药信息，属于受保护健康信息（PHI），HIPAA 法规适用。

**身份定位：** 当 MediSync 与医院或保险公司合作时，在 HIPAA 框架下属于"Business Associate"（业务关联方），必须签署 BAA（Business Associate Agreement），承担数据保护义务。

**最小必要原则：** 依据 HIPAA 的 Minimum Necessary Rule，MediSync 仅共享完成特定目的所需的最少信息。对保险公司，这意味着只提供聚合统计，不提供个人记录。

**合规即护城河：** 正因为 HIPAA 要求严格，MediSync 从产品架构设计之初就将个人数据和聚合报告分开处理。这不是妥协，而是合规驱动的产品设计决策。

### 4.4 AI 透明度与公平性

- **可解释性：** 用户可查看 AI 为何选择在某个时间提醒（如"根据您过去两周的记录，您通常在早餐后服药"）。
- **公平性测试：** 针对多样化人群（如轮班工人、跨时区旅行者、不规律作息者）进行测试，确保 AI 不会因为某类人群的行为模式较少见而降低提醒准确率。
- **临床安全兜底：** AI 不做任何临床决策。漏服重调度严格受药物安全窗口约束，窗口规则基于临床指南预设，非 AI 自主判断。

## 六、评估方案

### 6.1 研究设计

小规模试点研究，采用前后对比设计：

- **基线期（2周）：** 参与者使用标准固定时间提醒 APP。
- **干预期（2周）：** 参与者切换至 MediSync。

### 6.2 核心评估指标

- **按时服药依从率：** 对比基线期与干预期的按时服药比例（或使用 Proportion of Days Covered 指标）。
- **通知响应率：** 用户点击"已服药" vs 忽略/跳过的比例变化。
- **系统可用性评分：** 使用 System Usability Scale (SUS) 标准化量表。
- **用户满意度：** 定性访谈，了解用户对 AI 提醒时间选择、出门提醒、漏服重调度功能的感受。

### 6.3 预期展示

在答辩幻灯片中展示假设性对比框架，说明如何衡量 AI 引入前后的效果差异，即使没有真实数据，清晰的实验设计框架也能体现项目严谨性。
